# Codex 纠错记录

本文档记录用户已经纠正过的判断或操作规则。后续在 Toolaze web 项目中工作前，应优先检查并遵守这些记录，避免重复犯错。

## 2026-06-13

### 工作文件默认位置

- 错误判断：最初只创建了 `/Users/neva/Desktop/workspace`，但没有主动确保后续所有由 Codex 产生的文档、文件夹和项目都归入该目录。
- 用户纠正：以后凡是 Codex 产生的文档、文件夹等，都要放在 `/Users/neva/Desktop/workspace` 下。
- 后续规则：除非用户明确指定其他路径，否则所有 Codex 生成物默认放入 `/Users/neva/Desktop/workspace`；Toolaze web 相关资料默认放入 `/Users/neva/Desktop/workspace/toolaze-web/_codex`。

### Toolaze web 项目副本选择

- 错误判断：发现多个 `toolaze-web` 目录后，先把三个副本都移动到了 `workspace`，导致同名项目并列。
- 用户纠正：只应在 `workspace` 中保留最近有 git 提交的项目，其他同名副本移回原位置。
- 后续规则：遇到多个同名项目副本时，先比较 git 最新提交时间、远端和工作区状态；仅保留最明确的主副本，其他副本不得擅自合并或覆盖。

### prompts 与 toolaze-web 的关系

- 错误判断：一度把 `prompts` 当成独立项目或普通归档资料。
- 用户纠正：`prompts` 的起点是一类落地页，后来已经放入 Toolaze web 项目，它属于产品名 Toolaze 的 web 端项目。
- 后续规则：`prompts` 应视为 Toolaze web 的落地页历史工作区/素材源，而不是独立项目。其当前位置为 `/Users/neva/Desktop/workspace/toolaze-web/_codex/landing-pages/prompts`。

### 纠错记录维护

- 用户要求：凡是用户让 Codex 纠正的内容，都要记录下来，避免以后再犯同样的错。
- 后续规则：每次用户指出判断、路径、项目归类、实现方式或输出规则有误时，都要更新本文档；记录应包含错误判断、用户纠正和后续规则。

### 官方支持的模型能力可直接写入落地页

- 错误判断：在 GPT Image 2 SEO 策略中，把官方支持的能力也过度降级为“除非 Toolaze 当前生成器稳定支持并确认，否则不要写”，例如 4K output 和强文字渲染能力。
- 用户纠正：官方能支持的能力，Toolaze 页面也可以支持，这类内容应该直接写入 SEO 页面。
- 后续规则：写 Toolaze 模型落地页时，官方明确支持或官方文档确认的能力可以作为页面能力直接表达；只有当 Toolaze 产品策略明确不开放、有限制或用户要求谨慎时，才降级为条件性表述。

### 模型落地页首屏应优先呈现功能区

- 错误判断：为 GPT Image 2 页面新增了一个独立顶部 Hero 板块，功能生成器被放在第二个板块。
- 用户纠正：不要顶部第一个板块；功能区里面的内容就是 H1 和描述。
- 后续规则：Toolaze 模型落地页首屏应直接进入功能区；H1 和描述应放在真实生成器功能区内，避免先出现一个独立静态 Hero 再让用户下滑到工具。

### 本地预览必须确认样式资源可访问

- 错误判断：构建通过并给出本地预览地址后，没有重新确认浏览器实际加载的 CSS 资源是否仍然可访问。
- 用户纠正：预览页面没有样式。
- 后续规则：在运行 `npm run build` 后，如果本地 dev server 仍在运行，需要重启或使用 `npm run dev:clean` 重新启动预览服务；交付可预览页面前，应检查页面引用的 CSS 返回 `200 text/css`，避免旧 dev 进程与新的 `.next` 产物不同步导致样式 404。

### 不要把内部 SEO 关键词清单暴露到页面

- 错误判断：在 GPT Image 2 落地页中把 “Best entry queries” 这类 SEO 入口关键词列表做成了用户可见卡片。
- 用户纠正：这个模块没有必要。
- 后续规则：落地页可以围绕关键词组织内容，但不要把关键词研究过程、入口 query 列表、SEO 工作台式信息直接展示给用户；这些内容应保留在 `_codex/seo-strategies` 或内部文档中，页面只呈现对用户有实际帮助的解释、示例、对比和操作。

### 落地页图片不要滥用电脑屏幕场景

- 错误判断：为 GPT Image 2 落地页生成能力图时，默认使用显示器、电脑屏幕、工作台等实体场景来承载画面。
- 用户纠正：除非文字描述了电脑等实体场景需要，否则不要出现电脑屏幕中的图像，尽量用平面图。
- 后续规则：落地页配图应先匹配板块语义；没有明确实体设备场景时，优先使用平面海报、卡片矩阵、流程图、对比板、成品拼贴、信息图式画面，不用“屏幕里展示内容”作为默认表达。只有当文案明确讨论 UI mockup、dashboard、app screen 等设备/界面场景时，才可使用界面图，但也优先做平面 UI 画布而非电脑显示器照片。

### 平面图不等于卡通信息板，电商图必须有商用价值

- 错误判断：把“尽量用平面图”理解成统一生成卡通化、矢量化、信息板式图片，并且多张图重复使用保温瓶作为主体。
- 用户纠正：平面图不是都要卡通；电商相关图片应该是电商广告图、海报图、详情图等有商业使用价值的成品视觉，而且整页图片主体要多样化。
- 后续规则：生成 Toolaze 模型落地页图片时，应优先产出可直接作为商业样张的高质量视觉：产品广告图、海报图、详情页图、社媒广告图、包装图、教育信息图、UI 平面稿等。不要把所有图片做成同一种矢量卡片风格；不要在整页反复使用同一产品主体；同一页面要覆盖不同品类和场景，例如护肤品、饮料、耳机、鞋服、食品、SaaS UI、教育图表等。

### Prompt Examples 不应重复 Example Gallery

- 错误判断：在 GPT Image 2 页面中，`Best GPT Image 2 Prompt Examples` 使用了和 `Example Gallery for GPT Image 2` 基本相同的场景和图片，导致两个板块语义重复。
- 用户纠正：Prompt Examples 仍然需要图片，但提示词场景应基于模型擅长方向重新设计，不能和上面的 Gallery 重复。
- 后续规则：模型页的 Gallery 负责展示常见应用场景；Prompt Examples 负责展示可复制的能力型提示词，例如多语言文字、印刷约束、角色一致性、连续分镜、参考图编辑、专业版式等。两个板块可以都有图片，但图片和内容目标必须明显区分。

### 已认可的视觉风格不要随意改掉

- 错误判断：把用户之前认可的最终 CTA 紫色视觉改成了深色纯黑风格，破坏了用户偏好的页面收束效果。
- 用户纠正：`Create with GPT Image 2 on Toolaze for Free` 这个板块之前紫色很好看，应改回去。
- 后续规则：用户明确认可过的视觉处理应作为后续迭代的设计约束。除非有明确理由并先说明，否则不要随意替换成新的风格；修改时应优先在原风格基础上调整内容和资产。

### 外部社区来源板块要有媒体卡片

- 错误判断：为 GPT Image 2 页面添加 Reddit 和 X 外部讨论板块时，只做了纯文字链接卡片，没有像 Pollo 页面一样展示来源图片或视频缩略图。
- 用户纠正：X 和 Reddit 的用户模型链接板块也要放图，并且还要加类似 Pollo 的 YouTube 板块。
- 后续规则：模型落地页如果添加 Reddit、X/Twitter、YouTube 等外部来源区，应尽量做成媒体卡片：上方展示真实帖子图片、平台预览图或视频 thumbnail，下方展示标题、作者/来源、简短说明和外链。不要只放文字链接；无法稳定抓取真实预览图时，应明确使用可信的视觉兜底，避免伪造来源图片。

### X 社区板块应复刻 Pollo 的推文卡样式

- 错误判断：把 GPT Image 2 的 X 外部来源区做成了普通图片卡片，虽然有图和链接，但没有参考 Pollo 页面中推文嵌入卡的视觉结构。
- 用户纠正：X 板块要 1:1 复刻 Pollo 的板块样式。
- 后续规则：当模型落地页引用 X/Twitter 社区示例时，应优先使用推文卡片式布局：深色区背景、白色圆角卡片、头像、用户名、handle、Follow、X 标识、正文摘要、媒体预览、Watch on X、播放按钮、时间和互动行。不要只做普通内容卡。

### Reddit 预览图必须取原帖在线图片

- 错误判断：Reddit 图片加载失败后，把社区卡片图片替换成了 Toolaze 自己生成或本地语义替代图。
- 用户纠正：Pollo 上 Reddit 的图片地址是 `preview.redd.it`，应该直接拿 Reddit 在线图片地址，而不是放到自己的服务器上，也不要自己生成图片。
- 后续规则：Reddit 社区来源区必须优先使用原帖中的真实 `preview.redd.it` 图片地址；不要用自己生成的图替代 Reddit 原文图片。若某个 `preview.redd.it` URL 失效，应更换为可验证的同帖有效预览图，或更换帖子，而不是用非原帖图片冒充。本地预览中使用 `preview.redd.it` 时，图片标签应加 `referrerPolicy="no-referrer"`，避免 localhost Referer 导致热链被拒。

### 同一 Reddit 帖子多图不能用 href 做 React key

- 错误判断：Reddit 卡片用 `key={item.href}`，但同一个 Reddit 帖子拆成多张图片卡时，多个卡片的 href 相同。
- 用户纠正：页面报错 `Encountered two children with the same key`。
- 后续规则：社区媒体卡如果允许同一个帖子拆多张图，React key 必须使用唯一字段，例如图片 URL、显式 id、或 `href + image`，不能只用帖子 href。

### Reddit 图片地址必须取自原帖并验证能显示

- 错误判断：手写或猜测 `preview.redd.it` 图片 URL，只满足了“看起来像 Reddit 在线图”，但没有确认页面里真实加载成功。
- 用户纠正：图片还是没有显示。
- 后续规则：Reddit 社区卡必须从原帖页面实际暴露的媒体链接中取图，并在页面中检查图片加载状态；不能凭 slug、hash 或旧记录拼接 URL。若图片加载失败，应更换为同帖实际可用图片或更换帖子，而不是继续保留不可显示图片。

### Reddit CDN 直连失败时要先验证替代展示链路

- 错误判断：以为 `preview.redd.it` 链接来自原帖就一定能在本地页面显示，没有注意到当前网络对 `preview.redd.it` / `i.redd.it` 直连会返回 `ECONNRESET`。
- 用户纠正：图片仍然没有显示。
- 后续规则：如果 Reddit CDN 直连失败，必须保留原始 Reddit 图片来源字段，同时用已验证返回 `200 image/*` 的在线展示 URL 渲染卡片；修改前后都要批量检查 10 张图的 HTTP 状态、content-type 和非零文件大小。

### Reddit 多图帖不能拆成多个讨论卡片

- 错误判断：为了填满 Reddit 两行卡片，把同一个帖子里的多张图片拆成多个卡片，导致板块看起来像有很多不同来源，实际信息源重复。
- 用户纠正：Reddit 板块需要拿不同帖子；如果一个帖子里有多个图片或视频，应在网站卡片的媒体区提供左右切换按钮。
- 后续规则：社区来源区应以“一个帖子一张卡”为基本单位。多图片或视频帖子用卡片内轮播展示，保留一条外链指向原帖；不要把同一帖的多媒体拆成多张卡冒充多个讨论来源。

### 社区图片区不要给静态图加视频暗示

- 错误判断：X 卡片中的资源都是静态图片，却在图片区叠加了播放按钮，容易让用户误以为这是视频。
- 用户纠正：X 目前的资源都不是视频，不要给视频 icon；X 板块也要一排五个、两排。
- 后续规则：社区媒体卡必须按实际媒体类型展示。静态图只放图片和来源入口，不加播放三角形、视频徽标或其他视频暗示；需要 5x2 信息密度时，应提供 10 个卡片并使用 5 列桌面布局。

### 不要随便把界面文案强制全大写

- 错误判断：为了强化视觉层级，在 Reddit 来源、X 已认证标识、YouTube 标签、图片占位标签等位置使用了 `uppercase` 样式，导致普通文案被强制全大写。
- 用户纠正：之前已经规定过不要随便全大写；Reddit 群组名和 X 的已认证都变成纯大写不合适。
- 后续规则：除模型名、缩写、品牌官方写法、结构化数据字段等本来就应该大写的内容外，不要对用户可见 UI 文案使用 `uppercase`。平台名、来源名、状态标签、按钮和说明文字应保留自然大小写。

### 能力小卡不要和详细能力段落重复

- 错误判断：在模型落地页里先放 6 个 Key Features 小卡，再在下面用同样主题展开详细能力，导致结构重复。
- 用户纠正：删除小卡，把小卡标题放到下面详细能力段落上；详细能力作为 Key Features 的下级内容。
- 后续规则：如果能力点已经需要详细展开，就不要再做一组重复摘要小卡。保留一个 Key Features 主标题，下面用 H3 能力段落承接具体内容；除非小卡承担明确导航或对比功能，否则直接省略。

### 模型页标题字号和重点词要跟站内统一

- 错误判断：GPT Image 2 页面 H1/H2 使用了更大的响应式字号，没有跟 Toolaze 其他页面常用的 H1 40px、H2 36px 保持一致；H1 也没有用紫色强调核心关键词。
- 用户纠正：其他页面 H1 字体大小是 40，H2 是 36，且 H1 标题中会用紫色突出重点词。
- 后续规则：Toolaze 模型落地页默认 H1 使用 `text-[40px]`，H2 使用 `text-[36px]`；H1 中的模型名或核心关键词应使用站内紫色/靛紫渐变强调，除非已有更高优先级设计规范。

### Reddit 卡片需要平台标识

- 错误判断：Reddit 社区卡只有来源文字，没有在右上角显示 Reddit 平台标识。
- 用户纠正：reddit 右上角要有 reddit 的图标。
- 后续规则：社区来源卡在复刻平台视觉时，不能只展示帖子内容和标题，还要把平台识别元素放进卡片头部，例如 Reddit 标识、X 标识、YouTube 平台标签等，让用户一眼识别来源平台。

### X 卡片头像必须使用真实用户头像

- 错误判断：X 社区卡用用户名缩写作为头像占位。
- 用户纠正：twitter 要拿到用户实际的头像。
- 后续规则：引用 X/Twitter 真实用户帖子时，头像区域应使用公开可验证的真实用户头像 URL 或本地缓存后的真实头像，不要用缩写、假头像或泛用占位图。

### 社区板块顺序和 Reddit 密度

- 错误判断：YouTube 板块放在 Reddit 和 X 后面，Reddit 只做了少量大卡片。
- 用户纠正：YouTube 放到 Reddit 上面；Reddit 一行要五个，要两行。
- 后续规则：当用户要求类似 Pollo 的社区来源区时，应按用户指定的信息密度和顺序组织：视频讲解类 YouTube 可以放在 Reddit 讨论前；Reddit 卡片在桌面端应支持 5 列、2 行的紧凑展示，而不是只有 2 个大卡片。

### 工具页首屏结构要跟站内间距和背景统一

- 错误判断：把模型页面包屑放进首屏功能区内部，又给功能区加了较大的顶部 padding，导致面包屑距离顶部比其他 Toolaze 工具页更大；首屏背景也用了白色，和站内常见工具页的 `#F8FAFF` 不一致。
- 用户纠正：面包屑距离顶部偏大，页面背景色也有差异。
- 后续规则：Toolaze 工具页和模型功能页默认使用 `Navigation -> Breadcrumb -> main` 结构；首屏功能区背景优先保持 `#F8FAFF`，除非有明确设计理由。不要把面包屑放进带额外顶部 padding 的 hero 或工具容器里。

### 英文落地页文案不要写成内部流程说明

- 错误判断：首屏描述使用了 `in a simple browser workflow`，表达不够常见，也偏向产品内部流程描述。
- 用户纠正：觉得 `in a simple browser workflow` 不常见。
- 后续规则：面向用户的英文落地页应优先写直接、自然的价值表达，例如 `directly in your browser` 或 `right in your browser`；避免用 `workflow` 这类偏内部机制的词包装普通使用场景。

### 示例 Gallery 的数量要服务最终视觉排布

- 错误判断：`Example Gallery for GPT Image 2` 只放了 6 个示例，在桌面 4 列布局下会形成不完整的第二行，视觉密度和完整性不够。
- 用户纠正：Example Gallery 需要两排 8 个示例。
- 后续规则：如果 Gallery 使用桌面 4 列布局，应优先提供 8 个示例形成完整两排；示例数量要和最终网格排布匹配，不要只按内容刚够用来决定数量。

### Cloudflare Pages 静态导出页不要手写 `.html` clean URL 重写

- 错误判断：为了让 GPT Image 2 页面在线上可访问，在 `public/_redirects` 中把 `/model/gpt-image-2` 等 clean URL 重写到 `/model/gpt-image-2.html`。
- 用户纠正：Toolaze web 过去就是直接提交 Git 触发 Cloudflare 上线，发布后需要按线上实际访问结果继续排查，而不是停在“已提交”。
- 后续规则：Next.js 静态导出部署到 Cloudflare Pages 时，如果 `out/**/page.html` 已存在，优先依赖 Pages 的 clean URL 解析；不要为同一路由额外写到 `.html` 的 200 rewrite，否则可能触发 Cloudflare Pages 把 `.html` 规范化回 clean URL，造成 308 自循环。发布后必须用正式域名验证目标页面返回 200。

### 静态图片资源目录不能和页面路由同名

- 错误判断：把 GPT Image 2 页面图片放在 `public/model/gpt-image-2`，导致静态导出后同时存在页面文件 `out/model/gpt-image-2.html` 和资源目录 `out/model/gpt-image-2/`。
- 用户纠正：图片要传到服务器上去，但不能因此影响页面路由访问。
- 后续规则：页面配图要放在会随 Cloudflare Pages 发布的 `public` 目录中，但资源目录不能和页面 clean URL 同名。模型落地页图片优先放在 `public/model-assets/{model}`，页面中引用 `/model-assets/{model}/...`。

### 模型接入默认参数必须按模型要求设置

### Reddit 多图帖不能为了凑数量再次拆成多个帖子卡

- 错误判断：在 Seedream 4.5 Reddit 板块中又把同一个 Reddit 原帖里的多张图片拆成多个卡片，虽然 React key 改成了唯一 id，但页面仍然把一组图伪装成了多个讨论来源。
- 用户纠正：之前已经要求过 Reddit 需要拿不同帖子；如果一个帖子里有多张图或视频，要放在同一卡片里用左右切换，不要拆成好几个帖子。
- 后续规则：社区来源卡片的数据粒度必须是“来源帖子/线程”，不是“媒体文件”。一个 Reddit 原帖只能生成一个卡片；多张图片或视频放进该卡片的 `media` 数组，用左右按钮切换。不要为了满足 5 列 2 行或 10 条卡片的视觉密度目标而拆分同一个原帖；真实不同原帖不足时，应少展示或继续找不同原帖。该规则已补充到项目内 `pollo-model-seo-page` skill 和全局 SEO 落地页 skill。

### Reddit 社区来源不足时要继续找不同帖子并避开政治人物

- 错误判断：Seedream 4.5 Reddit 板块只保留了 4 个真实原帖就停止，导致板块密度明显不足；同时保留了一个搜索摘要里出现政治人物讨论的对比帖，不适合面向产品落地页展示。

- 后续规则：当页面目标是 Pollo 风格 5 列 2 行社区区时，应继续寻找真实、不同、可验证媒体能显示的 Reddit 原帖，不能只因前几条通过验证就停下。外部社区来源应过滤政治人物相关内容，优先使用非政治的创作者测试、提示词示例、产品/设计图、人像、风景、建筑、模型能力对比或 workflow 分享。

### 重复卡片必须等高且底部动作对齐

- 错误判断：X 社区卡正文长度不一，`Read X replies` 紧跟正文和元信息自然流动，导致同一行卡片底部按钮不在同一水平线。

- 后续规则：所有重复卡片网格都要用等高布局。正文区域最多显示四行并保留四行高度，超出使用省略号或 CSS line clamp；底部动作按钮/链接必须通过 flex 的 `mt-auto` 或等价布局对齐到底部。该规则已补充到项目内 `pollo-model-seo-page` skill 和全局 SEO 落地页 skill。

### Reddit 卡片必须验证标题、链接、媒体同源
